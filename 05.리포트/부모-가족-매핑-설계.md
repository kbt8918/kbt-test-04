# 부모-가족 매핑 및 호칭(관계) 설계

> 작성일: 2026-06-07
> 분류: 05.리포트 (구현 설계 참고 자료, NotebookLM 미등록)
> 관련 코드: `src/frontend` (Vercel 배포 대상)

## 1. 배경 / 문제 정의

기존 데이터 모델에서 부모님과 가족은 **하나의 가족 그룹(`groups`)에 함께 소속**되는 방식으로만 연결되어 있었다(`group_members` 연결 테이블). 이때 다음 한계가 있었다.

- "누가 부모님인지"는 `users.role = 'parent'` 로만 추론 가능하고, 가족 화면에는 멤버 목록을 불러오지 않아 항상 "부모님" / "홍길순(어머니)"로 **하드코딩**되어 있었다.
- 그룹 멤버 조회 API(`GET /api/groups/:id`)는 `role`만 내려줄 뿐 **이름·호칭(어머니/아버지)** 정보를 제공하지 않았다.
- 한 그룹에 부모님이 여러 명(아버지+어머니)일 때 이를 표현하거나 구분할 방법이 없었다.

## 2. 설계 결정

| 항목 | 결정 | 비고 |
|------|------|------|
| 부모 모델 | `role='parent'` 멤버를 **모두 부모님으로 인정(여러 명 허용)** | 아버지+어머니 동시 가능 |
| 호칭(관계) 저장 위치 | `group_members.relation` 컬럼 | 같은 사람이 그룹마다 다른 호칭 가능 |
| 호칭 지정 주체 | **가족이 그룹 내에서 지정** | 부모-자녀 직접 1:1 관계 테이블은 도입하지 않음 |

### 데이터 모델 (변경 후)

```
users (role: parent | family | admin)
  └─< group_members (group_id, user_id, role: owner|member, relation: 호칭) >─┐
groups ─────────────────────────────────────────────────────────────────────┘
```

- `relation`: 가족이 지정하는 호칭 라벨. 예) `어머니`, `아버지`, `할머니`. NULL 허용(미지정).
- "부모님 식별" = 그룹 멤버 중 `users.role = 'parent'` 인 사람. 호칭은 표시용 부가 정보.

## 3. 변경 범위

| 레이어 | 파일 | 변경 |
|--------|------|------|
| DB | `supabase/migrations/0002_group_member_relation.sql` | `group_members.relation varchar(20)` 추가 |
| 타입 | `lib/db.types.ts` | `GroupMembersRow.relation`, Insert/Update 반영 |
| API 조회 | `app/api/groups/[id]/route.ts` | 멤버 응답에 `name`, `relation` 포함 |
| API 수정 | `app/api/groups/[id]/members/route.ts` (신규) | `PATCH` 로 멤버 호칭 지정(그룹 구성원만) |
| 검증 | `lib/validation.ts` | `isValidRelation()` 추가 |
| 클라이언트 | `lib/apiClient.ts` | `groupDetail()`, `setMemberRelation()` 래퍼 |
| UI | `components/screens/family.tsx` | 실제 멤버·호칭 표시, 가족 설정에서 호칭 편집 |

## 4. 동작 흐름 (변경 후)

```
1. 가족이 그룹 생성 → 초대 코드 발급
2. 부모님이 초대 코드로 그룹 참여 (role=parent 로 가입돼 있음)
3. 가족 화면 > 가족 설정 > 구성원 목록에서
   부모님 멤버에 "호칭 지정"(어머니/아버지 등) → PATCH /api/groups/:id/members
4. 가족 지도/설정 화면이 이름 + 호칭으로 부모님을 표시
   (부모님이 2명이면 각각 이름·호칭으로 구분 표시)
```

## 5. 배포 주의사항

- `relation` 컬럼은 **NULL 허용**이라 기존 데이터와 하위 호환된다.
- 멤버 조회 API가 `relation` 컬럼을 SELECT 하므로, **코드 배포 전(또는 동시에) 상용 DB에 마이그레이션이 적용**되어 있어야 한다. 미적용 시 조회 쿼리가 실패한다.
- 적용 순서: ① Supabase에 `0002` 마이그레이션 실행 → ② 코드 배포(Vercel).
